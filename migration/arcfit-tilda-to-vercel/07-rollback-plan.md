# 07 — Rollback Plan (если что-то сломалось)

## Когда откатывать

| Симптом | Серьёзность | Действие |
|---|---|---|
| Сайт не открывается > 5 мин после DNS-переключения | **CRITICAL** | Немедленный rollback DNS |
| 5xx ошибки на главной | **CRITICAL** | Немедленный rollback DNS |
| Лид-форма не отправляет (заявки теряются) | **HIGH** | Rollback DNS, потом чинить /api/lead |
| Расписание показывает неактуальное | **MEDIUM** | НЕ откатывать, обновить FALLBACK_SCHEDULE и push |
| OG-теги/canonical не правильные | **LOW** | НЕ откатывать, фиксить в коде |
| Метрика не пишет | **MEDIUM** | НЕ откатывать, проверить env `NEXT_PUBLIC_YM_ID` |
| 301 не работают на 1-2 URL | **LOW** | НЕ откатывать, добавить в `next.config.ts` |

**CRITICAL = откат DNS немедленно. HIGH = откат если не починили за 30 мин. MEDIUM/LOW = чинить без отката.**

## Как откатить DNS (5 минут)

### 1. Зайти в nic.ru → arcfit.ru → DNS
### 2. Заменить A-record обратно

| Тип | Имя | Значение |
|---|---|---|
| A | `arcfit.ru` | **`76.76.21.21`** → **`185.215.4.30`** (Tilda) |
| CNAME | `www` | (вернуть прежнее значение из бэкапа) |

### 3. Сохранить
TTL=300 → новый старый IP виден через ~5-15 мин у большинства пользователей.

### 4. Удалить домен arcfit.ru из Vercel (опционально)
```bash
source ~/.claude/secrets/vercel.env
curl -X DELETE -H "Authorization: Bearer $VERCEL_TOKEN" \
  "https://api.vercel.com/v9/projects/$VERCEL_PROJECT_ID_ARCFIT_NEXT/domains/arcfit.ru?teamId=$VERCEL_TEAM_ID"
```
Не обязательно — Vercel домен без DNS-указания просто не работает.

### 5. Проверить
```bash
nslookup arcfit.ru
# 185.215.4.30 (Tilda)
curl -sI https://arcfit.ru/ | head -5
# server: nginx (Tilda) или server: Apache
```

## Что делать с лидами потерянными во время инцидента

Если в окно миграции пришли заявки и `/api/lead` упал — данных не будет. Поэтому:
- Перед миграцией предупредить контакт-центр: «возможна 30-минутная пауза»
- Если возможно — на странице повесить временный баннер «Сайт обновляется, свяжитесь по телефону +7 925 088 9196»
- После rollback — проверить TG/CRM — все ли заявки за период миграции есть

## Запасные варианты

### Если DNS-кеш у части пользователей застрял
- Прошёл TTL, но 5% всё ещё видят Vercel/Tilda по старому
- Это нормально, пройдёт за 1-24 часа
- Можно ускорить: попросить их `ipconfig /flushdns` (Windows), `sudo dscacheutil -flushcache` (Mac), `sudo systemd-resolve --flush-caches` (Linux)

### Если Tilda удалили / отключили
ЕСЛИ за время миграции Tilda была отключена в проекте (а так делать не надо до T+1 неделя!), то:
- Зайти в Tilda админку
- Вернуть публикацию
- Повторно нажать «Опубликовать все страницы»
- Подождать 2-5 мин, проверить https://arcfit.ru/ снова

### Если Vercel-проект сломан и Tilda отключена (одновременно)
Нужен запас:
- Перед миграцией — экспортировать Tilda как HTML+CSS архив (Tilda админка → Settings → Экспорт). Положить в `migration/tilda-export-2026-XX-XX.zip`.
- В крайнем случае залить этот архив на любой shared hosting (REG.ru, Beget) и временно указать DNS туда.

## Учёт инцидента

После rollback (если был):
1. Создать `migration/incidents/<дата>.md` с описанием:
   - Что сломалось
   - Симптомы
   - Действия
   - Время простоя
   - Root cause
2. Перенести RC-фиксы в код, обновить миграционный план
3. Назначить новую дату cutover

## Чего НЕ делать в панике

- ❌ Удалять Vercel-проект (теряются настройки, env vars)
- ❌ Откатывать git-commits на main (auto-deploy сразу выкатит более старую версию)
- ❌ Менять DNS на «угадайку» (произвольный IP) — будет offline
- ❌ Звонить в поддержку Tilda/Vercel в первую минуту — обычно на DNS-уровне можно решить за 5 мин

## После успешного rollback
- Tilda продолжает работать
- Vercel проект остаётся, ничего не потеряно
- Чинить корневую проблему в Next.js
- Когда исправлено — назначить повторное окно cutover
